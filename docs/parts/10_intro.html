第0章　はじめに ― 本書の読み方とプロジェクト概要

本書は、2D〇×（まるばつ）ローグライト「MaruBatsu」を、既存の 3DアクションゲームのフレームワークであるZankuuを土台として、ゼロから再構築できる ことを目的とした教育用ドキュメントです。各システムの設計思想・データの流れ・関数の役割を 解説し、続けて全ソースコードを掲載します。最終章では「どの順番で作るか」を示します。

0.1　本書の目的と対象読者

このプロジェクトの最大の特徴は、ゲームのルールそのもの（勝利条件・着手数・思考時間・ 駒の挙動など）を「フック」として抽象化し、アビリティで自由に差し替えられる 点にあります。単純な〇×ゲームではなく、「ルールが書き換わっていくローグライト」を成立 させるための設計が随所に盛り込まれています。

本書は次のような読者を想定しています。

  • C++とDirectXの基礎を学び終え、実際に動くゲームの設計を学びたい学習者
  • 本プロジェクトを引き継ぐ・保守する担当者
  • 「拡張しやすいゲーム設計」の具体例を知りたい開発者

読み進めるには、C++20の基本文法（クラス、継承、仮想関数、std::shared_ptr、 std::vector、ラムダ式）と、2Dスプライト描画の基本概念があると理解が早まります。 DirectXの初期化やウィンドウ管理といった「基盤」部分の内部実装は前提知識として扱い、 本書では深入りしません（0.3節参照）。

0.2　ゲームの全体像

MaruBatsuは、3×3の〇×ゲームを下敷きにしたローグライトです。 プレイヤーは〇、ボス（AI）は×を担当します。基本的な流れは次の通りです。

  1. ボス戦：制限時間（既定3分）の思考時間内に、〇×の対戦を行う。 縦・横・斜めのいずれかに自分の駒が3つ並べば勝利。
  2. 報酬選択：ボスを倒すと、3枚のカードからアビリティを1つ選んで獲得する。
  3. 次のボスへ：獲得したアビリティはランを通じて累積し、ゲームのルールを 少しずつ書き換えていく。これを繰り返してラスボスを目指す。

ボスもまた固有の「ギミック」を持ちます。たとえば最初のボス「氷盤の支配者」は、駒を 置くたびに盤上の全ての駒を一方向へ滑らせます。プレイヤーが獲得するアビリティと、 ボスのギミックは同じ「フック」という仕組みの上に乗っています。これが本設計の核心です。

プロト版のスコープ 本書が解説するコードは、コンテスト出展に向けたプロトタイプ版です。 ボスは1体（氷盤の支配者）、アビリティは6種＋ボスギミック1種を実装しています。 盤面サイズ拡張やレジェンダリー級アビリティなどはα版以降の予定として、 フックの拡張点だけを先に用意してあります。

0.3　「Zankuu基盤」とは何か

このプロジェクトは、3DアクションゲームZankuuのソースをコピーして出発しました。 3D特有のモジュール（モデル、3D当たり判定、カメラ等）は削除しましたが、 2Dゲームでもそのまま使える汎用基盤を流用しています。本書ではこれを 「Zankuu基盤」または「基盤層」と呼びます。基盤層が提供するものは次の通りです。

基盤層の機能主な提供物本書での扱い
ウィンドウ管理GameWindow_Create、ウィンドウプロシージャ前提（解説対象外）
Direct3D 11Direct3D_Initialize、バックバッファ、ブレンドステート前提（呼び出し方のみ）
2Dスプライト描画Sprite_Draw、Texture_Load前提（描画章で利用）
シェーダ2D用頂点／ピクセルシェーダ前提
入力Mouse、Keyboard、InputManager前提（利用方法は解説）
サウンドSoundManager前提
タイマーSystemTimer前提

つまり本書が解説する「ゲーム層」のコードは、これらの基盤APIを呼び出す側です。 「ゼロから作る」とは、Zankuu基盤という土台がある前提で、その上にゲーム層を一から 組み上げる、という意味になります。基盤層のAPIは、それぞれ ○○_Initialize / ○○_Update / ○○_Draw / ○○_Finalize という素直な 名前付けに従っているため、利用するだけなら難しくありません。

0.4　ディレクトリ構成と本書の用語

本書が解説する「ゲーム層」のファイルは、すべて game/ ディレクトリ配下と、 ルートの main.cpp です。構成は次の通りです。

main.cpp アプリのエントリーポイント（基盤層とゲーム層を接続） game/ ├ piece.h 駒・座標・陣営反転 ├ direction.h 4方向の定義 ├ game_state.h 1試合の状態スナップショット ├ board.h / .cpp 盤面データ（マス目と駒） ├ board_ops.h / .cpp 盤面操作（スライド処理） ├ win_check.h / .cpp 勝利条件の走査ロジック ├ collision.h / .cpp 2D衝突判定（汎用ユーティリティ） ├ ai.h / .cpp AI（ボスの思考） ├ run_state.h / .cpp 1ラン（複数戦）の進行状態 ├ render_primitives.h / .cpp 図形描画（線・矩形・円・×） ├ text_draw.h / .cpp テキスト描画（UTF-8対応） ├ font_atlas.h / .cpp 動的グリフアトラス（日本語対応） ├ fade.h / .cpp 画面フェード ├ ability/ │ ├ hooks.h フックインターフェイス群 ★最重要 │ ├ ability.h アビリティ基底クラス │ ├ ability_registry.* フック実装の集約レジストリ │ ├ default_hooks.* 標準ルール（フックのデフォルト実装） │ ├ ability_pool.* 報酬抽選用プール │ └ abilities/ 個々のアビリティ実装（7種） ├ boss/ │ ├ boss.h ボス基底クラス │ └ ice_board_boss.* ボス1「氷盤の支配者」 └ scene/ ├ scene.h / .cpp シーン切り替え管理 ├ title.h / .cpp タイトルシーン ├ game.h / .cpp ゲーム本編シーン ★中核 └ reward.h / .cpp 報酬選択シーン

本書で繰り返し使う用語を先に整理しておきます。

用語意味
基盤層 / Zankuu基盤Zankuuから流用した汎用部分（DirectX・入力・描画など）。
ゲーム層本書が解説する game/ 配下＋main.cpp。
フック (hook)ゲームのルールの差し替え点。インターフェイスとして抽象化されている。
アビリティ (ability)フックを実装し、ゲームのルールを書き換える効果。報酬で獲得する。

本書の構成 第1〜10章で各システムを「解説 → 全ソース掲載」の順に扱います。第11章では、 これらをどの順番で組み上げるかを構築手順としてまとめます。 まず通読して全体像を掴み、実際に作るときは第11章を地図として使う、という読み方を推奨します。
