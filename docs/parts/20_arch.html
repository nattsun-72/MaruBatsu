第1章　全体アーキテクチャ

本章では、アプリケーション全体の骨格を概観します。基盤層とゲーム層がどう接続され、 1フレームがどう処理され、3つのシーン（タイトル・ゲーム・報酬）がどう切り替わるかを 押さえます。個々のゲームロジックは第2章以降で詳述します。

1.1　レイヤ構成 ― 基盤層とゲーム層

このアプリは大きく2層に分かれます。下が基盤層（Zankuu由来の汎用機能）、 上がゲーム層（本書が解説する〇×ローグライト固有のコード）です。

┌─────────────────────────────────────────────┐ │ ゲーム層 (game/ + main.cpp) │ │ シーン（タイトル / ゲーム / 報酬） │ │ 盤面・フック・アビリティ・ボス・AI・UI │ ├─────────────────────────────────────────────┤ │ 基盤層 (Zankuu由来) │ │ ウィンドウ / Direct3D11 / スプライト描画 / 入力 │ │ サウンド / タイマー / フェード │ └─────────────────────────────────────────────┘

ゲーム層は基盤層のAPIを呼び出すだけで、基盤層がゲーム層を直接知ることは ありません（依存は一方向）。この分離のおかげで、ゲーム層を一から書き直しても基盤層は そのまま使えます。本書の「ゼロから作る」とは、この上半分を組み上げることを指します。

ほとんどの基盤モジュールは ○○_Initialize() / ○○_Update() / ○○_Draw() / ○○_Finalize() という統一された関数群を公開します。ゲーム層のシーンやサブシステムも、この命名規則に ならって作られています。「初期化・更新・描画・終了」の4点セットで考える、という方針を プロジェクト全体で一貫させているわけです。

1.2　メインループ (main.cpp)

main.cpp は唯一のエントリーポイントであり、基盤層とゲーム層を「接続する配線」です。 役割は3つに集約されます。

  1. 初期化：基盤層の各サブシステムを正しい順序で立ち上げ、最後にシーンを初期化する。
  2. メインループ：毎フレーム「更新 → 描画 → 表示 → シーン遷移確定」を繰り返す。
  3. 終了処理：初期化と逆順でサブシステムを片付ける。

初期化の順序には依存関係があります。たとえばスプライト描画やフォントアトラスは Direct3Dデバイスを必要とするため、Direct3D_Initialize の後でなければ 初期化できません。逆に終了処理は依存される側を後に片付けるため、初期化と逆順にします。

メインループ（WM_QUIT まで回る do-while）の1フレームは次の流れです。

PeekMessage でOSメッセージあり? ──Yes──→ Translate/Dispatch（ウィンドウ処理） │No ▼ [更新フェーズ] SystemTimer から経過時間を取得 KeyLogger_Update / PadLogger_Update ← 入力スナップショット更新 Scene_Update(elapsed_time) ← 現在シーンの更新 SpriteAnim_Update / Fade_Update [描画フェーズ] Scene_Draw() ← 現在シーンの描画 Fade_Draw() ← フェードを最前面に重ねる （DEBUGビルドのみ）FPS表示 Direct3D_Present() ← バックバッファを画面へ（VSync待ち） Scene_Refresh() ← 予約されたシーン遷移をここで確定

ポイントは、シーン遷移の確定（Scene_Refresh）をフレーム末尾で行うことです。 更新や描画の途中でシーンを切り替えてしまうと、「更新は旧シーン・描画は新シーン」のような 不整合が起きます。そこで遷移は「予約」だけにとどめ、1フレームの処理を完全に終えてから 切り替える設計になっています（詳細は次節）。

フレームレートは Direct3D_Present() の内部でVSync（垂直同期）に同期するため、 モニタのリフレッシュレートに追従します。経過時間 elapsed_time は SystemTimer から取得した実測値で、これを各 Update に渡すことで、 フレームレートに依存しない時間進行（タイマー消費・アニメーション）を実現します。

1.3　シーン管理 (scene)

本作には3つのシーンがあります。enum class Scene で表され、 同時に有効なのは常に1つだけです。

シーン役割
Scene::TITLEタイトル画面。入力でゲーム本編へ。
Scene::GAMEボス戦本編。盤面プレイ。
Scene::REWARDボス撃破後の報酬（アビリティ3択）画面。

scene.cpp は、現在シーン g_Scene と、次フレームの遷移先 g_SceneNext の2つの変数を持ちます。Scene_Update / Scene_Draw は g_Scene を見て、対応するシーンの関数 （Title_Update、Game_Draw など）へ switch で振り分けます。 これは「ポリモーフィズムを使わない素朴なディスパッチ」で、シーン数が少ないうちは これで十分シンプルに保てます。

遷移は「予約 → 確定」の2段階

遷移の流れは次の通りです。

どこかで Scene_Change(Scene::GAME) を呼ぶ → g_SceneNext = GAME（予約するだけ。まだ切り替わらない） フレーム末尾で Scene_Refresh() → g_Scene != g_SceneNext なら: SoundManager_StopAll() 旧シーンの音を止める Scene_Finalize() 旧シーンの終了処理 g_Scene = g_SceneNext 切り替え Scene_Initialize() 新シーンの初期化

この2段階により、シーンの Finalize と次シーンの Initialize が 必ずフレームの境界で実行されることが保証されます。Game_Update の中で Scene_Change(Scene::REWARD) を呼んでも、その瞬間にゲームシーンの後片付けが 走ることはなく、安全です。

描画前のレンダーステート設定 Scene_Draw は各シーンの描画に先立ち、バックバッファをレンダーターゲットに 設定し、深度テストを無効化（2Dなので不要）、アルファブレンドを有効化、 Sprite_Begin() を呼びます。各シーンはこの「2D描画の準備が済んだ状態」を 前提に描画コードを書けます。
