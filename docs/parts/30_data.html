第2章　基礎データ構造

ゲームのほぼ全モジュールが参照する、最小単位のデータ型を定義します。 「駒」「座標」「方向」「試合状態」の4つです。ここを最初に固めることで、 以降のシステムは共通の語彙で書けるようになります。

2.1　駒と座標 (piece.h)

piece.h は3つの要素を定義します。いずれもヘッダのみ（実装ファイル不要）です。

  • enum class Piece ― 1マスの状態。 Empty（空き）／Player（プレイヤーの〇）／ Enemy（ボスの×）の3値。盤面はこの値の集合として表現されます。
  • struct Vec2 ― int x, y の2D整数座標。 盤面のマス座標に使います。x は列（横）、y は行（縦）。 負値（例 {-1,-1}）を「無効座標」の意味で使う慣習があります。
  • Opponent(Piece) ― 相手陣営を返すヘルパ。 Player⇔Enemy を反転します。ターン交代で多用します。

ここで重要なのは、Piece が「駒の所有者」と「マスの状態」を兼ねている点です。 空きマスも1つの Piece 値（Empty）として扱うことで、盤面を 「Piece の配列」というシンプルな形で表現でき、特別扱いが減ります。

2.2　方向 (direction.h)

direction.h は上下左右の4方向 enum class Direction と、 方向→数値ベクトルへの変換ヘルパを提供します。氷盤（全駒スライド）、氷駒（1マス滑走）、 方向インジケータの矢印描画など、「方向」を扱うすべての箇所で共通利用します。

  • DirX(d) / DirY(d) ― 方向のX成分・Y成分（-1 / 0 / +1）。 座標 (x,y) に足せば1マス先が得られます。
  • DirectionName(d) ― 「上／下／左／右」の文字列。UI表示用。
画面座標系に注意 DirY は Up が −1、Down が +1 を返します。これは「画面の上ほどY座標が 小さい」という2Dスクリーン座標系に合わせた定義です。盤面のマス座標もこの向きに統一して おくと、描画とロジックの座標がそのまま一致し、変換ミスが減ります。

2.3　試合状態 (game_state.h)

GameState は、1試合の状態を丸ごと1つにまとめた構造体です。 盤面・手番・残り思考時間・着手数・決着情報・効果アニメーション情報を持ちます。

この構造体が重要なのは、フック（第4章）に渡される共通の引数だからです。 勝利条件・設置イベント・ターン処理といった各フックは、この GameState を 受け取り、必要に応じて参照・書き換えを行います。「試合に関する情報は全部ここにある」 という1か所に集約することで、フックの引数がシンプルになり、新しいフックを足すときも 迷いません。

メンバ意味
board盤面データ（第3章）。
currentPlayer現在の手番（Player か Enemy）。
remainingTime現ターンの残り思考時間（秒）。0以下で時間切れ。
turnCount経過ターン数。
placementsRemaining現ターンに「あと何個置けるか」。通常1、二手打ちで2。
result試合結果（MatchResult）。進行中／勝利／敗北／引分／時間切れ。
winner決着時の勝者。
animPhases効果アニメーションのフェーズ列（第10章で詳述）。

MatchResult 列挙型は、試合の進行状態と決着種別を1つにまとめています。 Playing 以外の値になった瞬間、試合は決着しています。 animPhases は「駒を動かす効果」がアニメ用の移動記録を積むための領域で、 基礎データのうちでは唯一やや特殊です。これは第10章「効果アニメーション」で扱います。
