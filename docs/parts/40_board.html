第3章　盤面システム

〇×ゲームの「土俵」である盤面を扱うシステムです。盤面データそのもの、駒を動かす スライド操作、勝利条件の走査、そして汎用の衝突判定を解説します。 ここはアビリティやボスギミックから繰り返し呼ばれる土台になります。

3.1　盤面データ (board)

Board クラスは、width × height のマス目を保持します。 内部表現は1次元の std::vector<Piece> で、座標 (x,y) は index = y * width + x で1次元インデックスに対応づけます（Index()）。 2次元配列ではなく1次元 vector にしているのは、サイズ可変（Resize）を 素直に扱うためです。

境界外アクセスの安全化

Board の設計でとくに大事なのは、Get と Set が 盤外座標を安全に扱う点です。

  • Get(x,y) ― 盤外なら Piece::Empty を返す。
  • Set(x,y,p) ― 盤外なら何もしない（無視）。

この「盤外＝空マス扱い」のおかげで、勝利条件の走査やスライド処理が、 わざわざ境界チェックを書かなくても Board::Get を呼ぶだけで安全に書けます。 たとえば後述の win_check は「1マス手前が同色か」を Get(x-dx,y-dy) で 調べますが、手前が盤外でも Empty が返るので破綻しません。境界処理を Board の中に1か所だけ閉じ込める、という考え方です。

サイズ変更 (Resize)

Resize(w,h) は盤面サイズを変えて全マスを空に戻します。これは「盤面拡張」 アビリティ（第5章）が、試合開始時に3×3を4×4へ広げるために使います。 盤面サイズを固定値でハードコードせず、最初から可変にしてあるのがポイントです。

3.2　盤面操作 ― スライド (board_ops)

BoardOps 名前空間は、駒を移動させる「純粋な盤面操作」をまとめます。 これらは氷盤（ボスギミック）や氷駒（アビリティ）から共通で呼ばれます。

移動の記録 ― Move / MoveList

スライド関数は、盤面を書き換えるだけでなく「どの駒がどこからどこへ動いたか」を 記録できます。これが BoardOps::Move（from座標・to座標・駒種）であり、 その集まりが MoveList です。1つの MoveList ＝ 同時に起きる移動の集合 ＝ アニメーションの1フェーズ、という対応になっています。盤面を一瞬で書き換えても、 この記録があれば「駒が滑っていく様子」を後からアニメーション再生できます（第10章）。

SlideAll ― 全駒スライド（2048タイル風）

SlideAll(board, dir, outMoves) は、盤上の全ての駒を指定方向の端まで 詰めます。アルゴリズムは行（または列）ごとに次の3段階です。

  1. そのライン上の非空マスの駒を、出現順を保ったまま収集する。
  2. ラインを一旦すべて空にする。
  3. 端から詰め直して再配置し、実際に位置が変わった駒を outMoves に記録する。

「左右スライドなら行ごと・上下スライドなら列ごと」「右下方向なら末尾寄せ・ 左上方向なら先頭寄せ」を、horizontal と toEnd の2つの真偽値と cellAt ラムダで切り替えています。4方向それぞれに別ループを書かず、 1つのロジックでまとめているのが読みどころです。

SlideOne ― 1マスだけ滑らせる

SlideOne は指定マスの駒を1マスだけ動かします。滑り先が 盤内かつ空マスのときのみ移動し、移動できたかを bool で返します。 氷駒アビリティ（駒1個を1マス滑らせる）が、これを所持数ぶん繰り返して使います。

3.3　勝利条件の走査 (win_check)

WinCheck 名前空間は、「縦・横・斜めに駒が並んでいるか」という純粋な走査 ロジックを提供します。勝利条件フックのデフォルト実装やAIの先読みが、これを利用します。

HasLine ― N並びの判定

HasLine(board, player, required) は、player の駒が required 個（既定3）連続している箇所があるかを返します。実装の要点は2つです。

  • 走査方向は4つだけ：右・下・右下・右上。左・上・左下・左上は 逆走査になるだけなので省略できます。
  • 始点からのみ数える：「1マス手前が同色なら、そのラインは別の始点で 数えるのでスキップ」という枝刈りを入れ、同じラインを重複して数えません。

盤面サイズに依存しない汎用実装なので、3×3でも4×4でも、また将来「4並びで勝利」に ルールが変わっても、required を変えるだけで対応できます。

CountMaxLine ― 最長連続数

CountMaxLine は、ある陣営の「最も長く繋がっている駒数」を返します。 AIが盤面を評価する材料などに使えます。

3.4　衝突判定 (collision)

collision は、円・矩形（AABB）同士の重なり判定と、点の内外判定を提供する 汎用2Dユーティリティです。Zankuuから流用し、2D向けにスリム化したモジュールです。

本作のUIヒット判定（ボタンやカードのクリック）は、実際にはこのモジュールを介さず 各シーン側で直接矩形比較していますが、汎用部品として用意されています。 円判定が「距離の二乗で比較して sqrt を避ける」など、2D衝突判定の 基本テクニックを学ぶ題材にもなります。
