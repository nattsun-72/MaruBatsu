第4章　フック設計 ― 本プロジェクトの心臓部

本章は本書で最も重要です。「ゲームのルールそのものを差し替え可能にする」フック設計を 解説します。この仕組みがあるからこそ、アビリティでルールが書き換わるローグライトが 無理なく成立します。逆に、ここを後付けしようとすると全面的な作り直しになります。

4.1　なぜフックなのか

普通に〇×ゲームを作るなら、勝利判定は「3並びがあるか」と直接書き、思考時間は 「180秒固定」と直接書くでしょう。しかし本作はローグライトです。アビリティによって 次のようなことが起こります。

  • 思考時間が +30秒される（熟考）／ −90秒される（呪い系）
  • 1ターンに2回置けるようになる（二手打ち）
  • 盤面が3×3から4×4へ広がる（盤面拡張）
  • 駒を置くと盤上の駒が滑る（氷盤・氷駒）
  • 時間切れの敗北が打ち消される（不死／予定）

これらをすべて「if (氷盤を持っている) { ... }」のような分岐でゲーム本体に 書き込むと、アビリティが増えるたびにゲームループが膨れ上がり、組み合わせも破綻します。

フック設計の核心 そこで本作は発想を逆転させます。「勝利条件」「設置イベント」「ターン処理」「思考時間」 「敗北処理」「盤面サイズ」というルールの構成要素を、それぞれインターフェイス（フック） として抽象化します。ゲーム本体はルールを直接書かず、「現在登録されているフック」に 判断を委ねます。アビリティはフックの実装を差し替える／追加することでルールを書き換えます。

この設計は後付けが効きません。プロト版の最初期、まだアビリティが1つも無い段階から フックを通してゲームを動かしておく必要があります。デフォルト実装（4.4節）が「アビリティ ゼロの標準ルール」を担うため、フックを通しても素の〇×ゲームとして問題なく動きます。

4.2　フックインターフェイス群 (hooks.h)

hooks.h は6つのインターフェイス（純粋仮想クラス）を定義します。 これがゲームルールの「差し替え点」の一覧です。

フック役割呼ばれるタイミング
IWinCondition勝利条件の判定着手後の評価時
IPlacementHandler駒設置時のイベント駒を1個置いた直後
ITurnHandler1ターンの着手数・ターン境界処理ターン開始/終了時
ITurnTimeProvider思考時間の決定ターン開始時
IDefeatHandler敗北の打ち消し敗北が発生したとき
IBoardModifier盤面サイズの決定試合開始時

各フックの考え方を補足します。

  • IWinCondition::Check ― 盤面と陣営を受け取り、勝っているかを返す。 「3並び」も「隅の支配」も、この実装の違いとして表現できます。
  • IPlacementHandler::OnPlace ― 駒が置かれた直後に呼ばれ、 GameState を書き換えられる。氷駒・氷盤がここで駒を滑らせます。 盤面を動かす効果はここに集まります。
  • ITurnHandler ― 1ターンの着手数（GetPlacementCount）と、 ターン開始/終了の処理を持つ。二手打ちは着手数を2にします。
  • ITurnTimeProvider::GetTurnTimeSeconds ― その陣営の制限時間を返す。 熟考・余裕が加算します。
  • IDefeatHandler::OnDefeat ― 敗北発生時に呼ばれ、 true を返すと敗北を打ち消す。不死などの「敗北回避」系の受け皿です。
  • IBoardModifier::GetBoardSize ― 盤面の縦横マス数を決める。 盤面拡張が3×3を4×4にします。

4.3　レジストリ (ability_registry)

AbilityRegistry は、「現在有効なフック実装」を集約し、ゲーム本体へ提供する 窓口です。ゲームシーンはフックを直接触らず、すべて AbilityRegistry 経由で 呼び出します。

単一フックと複数フック

フックは保持の仕方で2種類に分かれます。

種別該当フック保持方法
単一（差し替え式） IWinCondition / ITurnHandler / ITurnTimeProvider / IBoardModifier shared_ptr を1つ。Set○○ で上書き。
複数（追加式） IPlacementHandler / IDefeatHandler vector。Add○○ で追加し、全部呼ぶ。

「勝利条件」や「思考時間」は本質的に1つの答えを出すものなので単一にします。 一方「設置イベント」は、氷駒と氷盤が同時に効くなど複数が共存しうるので、 登録順に全部呼びます。「敗北処理」も複数の打ち消し手段が共存しうるため追加式です。

呼び出しヘルパ

CheckWin / OnPlace / GetPlacementCount / GetTurnTime / HandleDefeat / GetBoardSize などの ヘルパが、保持しているフックへ処理を委譲します。ゲーム本体から見ると、 g_Registry.CheckWin(board, player) のように1行でルールを問い合わせられます。

4.4　デフォルト実装 (default_hooks)

default_hooks は、アビリティを1つも持っていないときの「標準ルール」です。 AbilityRegistry は構築時とリセット時に、これらをセットします。

デフォルト実装標準ルール
DefaultWinCondition縦横斜めの3並びで勝利。
DefaultTurnHandler1ターン1着手。ターン境界処理なし。
DefaultTurnTimeProvider制限時間180秒。
DefaultBoardModifier盤面サイズ3×3。

デフォルト実装があるおかげで、「フックを通す」こと自体は素の〇×ゲームの動作を 何も変えません。フック設計の足場を最初に組んでも、ゲームは普通に遊べます。

4.5　チェインの仕組み

単一フックは1つしか保持できません。では、熟考（+30秒）と余裕（+60秒）を両方持ったら どうなるのでしょうか。答えがチェインです。

AbilityRegistry::SetTurnTimeProvider は、新しい実装をセットすると同時に 古い実装を戻り値で返します。アビリティはこの「旧実装」を自分の中に chained として保持します。そして自分が呼ばれたら、まず chained に 問い合わせ、その結果に自分の効果を足して返します。

登録の流れ（熟考 → 余裕 の順に登録した場合） registry 初期状態: [Default(180秒)] 熟考.RegisterTo: chained ← registry.SetTurnTimeProvider(熟考) // 旧 Default を受け取る registry: [熟考] ── chained ──> [Default] 余裕.RegisterTo: chained ← registry.SetTurnTimeProvider(余裕) // 旧 熟考 を受け取る registry: [余裕] ─> [熟考] ─> [Default] 呼び出し: registry.GetTurnTime(Player) 余裕.Get(): 熟考.Get() の結果 + 60 熟考.Get(): Default.Get() の結果 + 30 Default.Get(): 180 → 210 → 270

このように、単一フックでも連結リスト状に積み重なって全効果が合算されます。 複数フック（設置イベント等）は vector に並べて全部呼ぶだけなので、 チェインは不要です。第5章以降の各アビリティで、この chained パターンが 繰り返し登場します。

登録順が意味を持つ チェインは登録順に積まれます。たとえば設置イベントでは、プレイヤーのアビリティを 先に登録し、ボスのギミックを後に登録します。こうすると OnPlace は 「プレイヤー効果 → ボス効果」の順で発火します（氷駒で置いた駒を滑らせてから、 氷盤で全体を滑らせる、という意図した順序）。登録順は第10章の LoadBoss で確定します。
