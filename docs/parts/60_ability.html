第5章　アビリティシステム

第4章のフックを実際に「実装する側」がアビリティです。本章では、全アビリティの共通基底、 報酬抽選プール、そしてプレイヤーが獲得する6種のアビリティを1つずつ解説します。 各アビリティは「どのフックを、どう実装するか」の具体例になっています。

5.1　アビリティ基底クラス (ability.h)

Ability は全アビリティの共通基底です。表示情報（名前・説明・レアリティ）と、 2系統の振る舞いを定義します。

パッシブ系とアクティブ系

  • パッシブ系 ― フックを実装してルールを書き換える。 RegisterTo(registry) で自身をレジストリに登録します。 熟考・氷駒・二手打ち・盤面拡張がこれです。
  • アクティブ系（任意発動） ― activatable = true とし、 ゲーム中にボタンとして表示され、クリックで Activate が呼ばれる。 CanActivate（発動可否）と ChargesLeft（残り回数）も持ちます。 集中がこれです。

1つのアビリティがパッシブとアクティブを兼ねることも可能ですが、プロト版の6種は どちらか一方です。アクティブ専用のアビリティは、フック登録が不要なので RegisterTo を空実装にします。

shared_from_this とフック登録

Ability は std::enable_shared_from_this を継承しています。 アビリティは「Ability としての顔」と「フック（例 ITurnTimeProvider） としての顔」を多重継承で併せ持つため、登録時には自分自身をフック型の shared_ptr としてレジストリへ渡す必要があります。 shared_from_this() で自分の shared_ptr を取得し、 dynamic_pointer_cast で目的のフック型に変換して登録する、という手順を 各アビリティの RegisterTo が踏みます。

レアリティ

enum class Rarity は コモン／レア／エピック／レジェンダリー／呪い の5段階。 報酬カードの枠色や、将来の抽選重み付けに使います。

5.2　抽選プール (ability_pool)

AbilityPool は、報酬画面の3択を生成するための候補リストです。

  • CreateAll() ― 報酬候補となる全アビリティを、毎回新規インスタンスで 生成して返す。インスタンスを使い回さないのは、アビリティが chained やチャージなどの状態を持つためです。
  • PickRandom(count) ― 全候補をシャッフルし、先頭 count 個を返す （重複なし抽選）。報酬画面はこれで3つ取り出します。

プロト版は全候補を等確率で抽選しますが、α版以降ではレアリティ重み付き抽選へ 拡張する想定です。その差し替えも、この PickRandom 1か所で済みます。

5.3　各アビリティの実装

熟考 (ContemplationAbility) ― 思考時間 +30秒

ITurnTimeProvider を実装するパッシブアビリティです。RegisterTo で SetTurnTimeProvider を呼び、戻り値（旧実装）を chained に保持します。 GetTurnTimeSeconds は、対象が targetSide なら 「chained の結果 ＋ bonusSeconds」を返し、それ以外は chained の結果をそのまま返します。第4章で図解したチェインの典型例です。

このクラスは「余裕」の基底も兼ねており、加算量を変えるだけで派生が作れるよう、 bonusSeconds をメンバ変数にしてあります。

余裕 (MarginAbility) ― 思考時間 +60秒

ContemplationAbility を継承し、コンストラクタで名前・説明・レアリティ・ bonusSeconds を上書きするだけの派生クラスです。ロジックは一切書きません。 「基底クラスのパラメータ違い」で新アビリティを増やせる、という設計の好例です。

集中 (FocusAbility) ― 任意発動：このターン +2分

唯一のアクティブ系アビリティです。activatable = true とし、 フックは実装しません（RegisterTo は空）。代わりに、

  • Activate(state) ― state.remainingTime に bonusSeconds（120秒）を 加算し、charges を1減らす。
  • CanActivate(state) ― charges > 0 なら発動可。
  • ChargesLeft() ― 残りチャージ数を返す（UIの回数表示に使う）。

パッシブが「常に効くルール変更」なのに対し、アクティブは「プレイヤーが任意の タイミングで切る、回数制限つきの効果」です。ゲームシーンのHUDがこのボタンを描画し、 クリックで Activate を呼びます（第10章）。

氷駒 (IceTileAbility) ― 置いた駒が1マス滑る

IPlacementHandler を実装するパッシブアビリティです。RegisterTo では AddPlacementHandler を使います（設置イベントは複数登録できる追加式フック）。

OnPlace は、置かれた駒が自陣営なら、その駒を滑走方向へ1マス滑らせ、 移動を animPhases に1フェーズとして記録します。注目すべきは 重ね掛け対応です。氷駒を複数所持すると OnPlace が所持数ぶん 呼ばれます。各回で「設置マスから滑走方向へ走査して駒の現在位置を特定 → そこから 1マス滑らせる」ことで、先に滑った結果を引き継ぎ、合計で所持数ぶんのマスを滑ります。 複数インスタンスを特別にマージする仕組みを持たず、各インスタンスが素直に協調動作する のがうまい点です。

二手打ち (TwoHandsAbility) ― 1ターンに2回置ける

ITurnHandler を実装するパッシブアビリティです。ITurnHandler は 単一フックなので、SetTurnHandler でチェインします。 GetPlacementCount は、対象陣営なら2、それ以外は chained へ委譲。 OnTurnStart / OnTurnEnd もチェイン先へ委譲するだけです。

「自分の関心事（着手数）だけ変え、それ以外はチェイン先にそのまま流す」という、 チェイン型フックの定石どおりの実装です。

盤面拡張 (BoardExpandAbility) ― 盤面を3×3から4×4へ

IBoardModifier を実装するパッシブアビリティです。GetBoardSize は chained が決めたサイズを基準に、自分の目標サイズ（4×4）との大きい方を 採用します（縮小はしない）。試合開始時にゲームシーンがこのフックを呼び、 得られたサイズで Board::Resize します。

盤面が広がっても勝利条件は3並びのままなので、勝ち筋（フォーク）が大幅に増える 強力なエピックアビリティになります。盤面サイズという「構造」を変える効果も、 他と同じフックの枠組みで表現できることを示す好例です。
