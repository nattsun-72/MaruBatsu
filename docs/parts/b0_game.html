第10章　ゲームシーンの詳解

ゲームシーン（game.cpp）は本作の中核であり、最大のファイルです。盤面プレイ、 フックの統合、AI、効果アニメーション、HUD、アビリティUIのすべてがここに集まります。 本章ではコードの流れを6つの観点に分けて解説し、最後に全ソースを掲載します。

10.1　状態と定数

game.cpp は、無名名前空間にレイアウト定数・配色・永続状態をまとめています。 ゲームシーンが保持する主な状態は次の通りです。

状態役割
g_State試合状態（GameState）。盤面・手番・残り時間など。
g_Registryフック実装の集約（AbilityRegistry）。
g_Boss / g_BossAbilities現在のボスと、そのギミック。
g_AiTimer / g_AiPendingAI着手の遅延制御。
g_AnimPhases ほか効果アニメーションの再生状態（10.4節）。
g_AbilityPopupOpen ほかアビリティ履歴ポップアップの状態（10.6節）。

盤面は画面中央に固定サイズ（BOARD_SIZE）で描かれます。1マスの大きさ CellSize() は、盤面のマス数で BOARD_SIZE を割って動的に求めます。 盤面拡張で3×3が4×4になっても、この1か所で吸収できる仕組みです。

10.2　試合の初期化フロー

1試合の準備は ResetMatch() が行います。流れは次の通りです。

ResetMatch() ① g_State を初期化（盤面リセット、結果=Playing、ターン1） ② g_Registry.ResetToDefaults() フックを標準ルールへ戻す ③ 演出・アニメ・ポップアップ状態を初期化 ④ LoadBoss() ・ボスを生成 (IceBoardBoss) ・プレイヤーのアビリティを先に RegisterTo ┐ この順序が ・ボスのギミックを後に RegisterTo ┘ 発火順を決める ⑤ 盤面サイズフックを適用 int w=3,h=3; g_Registry.GetBoardSize(w,h); board.Resize(w,h); ⑥ BeginTurn(Player) 先手プレイヤーのターン開始

ステップ④の登録順が重要です。プレイヤーのアビリティを先に登録するため、 チェイン（単一フック）ではボス側が外側のラッパになり、設置イベント（複数フック）では プレイヤー効果が先に呼ばれます。第4章・第6章で触れた「氷駒で滑らせてから氷盤で滑らせる」 順序が、ここで確定します。

BeginTurn(p) は、その陣営の着手数と思考時間をフック経由で取得します （GetPlacementCount / GetTurnTime）。つまり二手打ちや熟考の効果は、 ターン開始時に自動で反映されます。敵のターンならAI着手を予約します。

10.3　着手と評価のフロー

駒を1個置く処理は、プレイヤー入力（HandlePlayerInput）とAI（HandleAiTurn）の 両方から、共通の PlaceAndEvaluate(pos, placedBy) へ集約されます。

PlaceAndEvaluate(pos, placedBy) ① 空きマスでなければ中断 ② 盤面に駒を置く、placementsRemaining を1減らす ③ 設置グロー演出の位置を記録 ④ animPhases をクリア、設置直後の盤面を boardStart として控える ⑤ g_Registry.OnPlace(...) 設置イベントフック発火 （氷駒・氷盤がここで animPhases に移動を積む） ⑥ StartEffectAnimation(boardStart) 積まれたフェーズを再生用に展開 ⑦ EvaluateAfterPlace(placedBy) 勝敗・ターン進行を評価

EvaluateAfterPlace は、勝利 → 引き分け → 追加着手 → ターン終了 の順で判定します。 勝利判定で押さえるべき点が1つあります。

両陣営の勝敗を確認する 氷盤ギミックは全駒を動かすため、「着手していない側」の駒が並ぶこともあります （ボスが置いた→スライド→プレイヤーの〇が斜めに揃う、など）。そのため勝利判定は、 着手側だけでなくプレイヤー・ボスの両方について g_Registry.CheckWin を呼びます。双方が同時に揃った場合はプレイヤーを優先します （ボス側のランダムなギミックによる理不尽な敗北を避けるため）。

10.4　効果アニメーション

氷盤や氷駒は盤面を一瞬で書き換えますが、それをそのまま表示すると駒が「パッと」 動いて分かりにくくなります。そこで本作はフェーズ方式の効果アニメーションを 採用します。

フェーズという単位

盤面を動かす効果フックは、自分の移動を GameState::animPhases へ 1つの MoveList（＝1フェーズ）として積みます。1フェーズ内の移動は同時、 フェーズ間は順番に再生されます。たとえば氷駒2個＋氷盤なら、 「氷駒の1マス移動」「氷駒の1マス移動」「氷盤の全駒移動」の3フェーズが積まれ、順に流れます。

再生の仕組み

StartEffectAnimation は、各フェーズ開始時点の盤面を g_PhaseStartBoards へ前計算します。Game_Update はアニメ再生中、入力・AI・タイマーを止めて フェーズを順に進めます。DrawBoardAndPieces は、再生中フェーズについて、 動かない駒は静止描画し、動く駒は from→to の補間（イーズアウト）で描きます。 これにより、駒が滑っていく様子が滑らかに見えます。

設計の指針 「効果が発動したら必ずアニメーションで視覚化する」のが本作の方針です。フェーズ方式は 特定の効果に依存しないため、新しい「駒を動かす効果」を足しても、その効果が animPhases へ移動を積みさえすれば、再生側は無改修で対応できます。

10.5　更新ループと入力

Game_Update は、状態に応じて3つのモードに分岐します。

Game_Update(dt) ・演出クロック更新、マウスホイール変化量を算出 ・R キーでリスタート（ESCはウィンドウ側の終了確認に割当て済みのため不使用） ┌ 効果アニメ再生中 ──────────────────────── │ 入力/AI/タイマーを止め、フェーズを順に進めるだけ ├ プレイ中 (result == Playing) ───────────── │ HandleTurnTimer（思考時間を消費） │ ポップアップ表示中? │ Yes → HandleAbilityPopup（盤面入力は止めるが時間は進行） │ No → HandleAbilityClick → 消費しなければ HandlePlayerInput │ HandleAiTurn └ ゲーム終了後 ────────────────────────── 一定時間待って、結果画面のボタンをマウスで処理

入力は「アビリティUIのクリックを先に判定し、消費しなければ盤面クリックへ回す」という 優先順で処理します。これにより、アビリティボタンの上をクリックしても誤って駒が 置かれません。ESCキーはウィンドウ側で終了確認ダイアログに割り当て済みのため、 リスタートやタイトルへの遷移はすべてマウス操作で完結するよう設計されています。

10.6　HUDとアビリティ履歴UI

DrawHud は、残り時間・手番・ボス名・所持アビリティを描画します。 所持アビリティ表示には、やや込み入った仕様があります。

  • グルーピング ― 同名アビリティは「氷駒 ×3」のようにまとめて表示。 BuildAbilityGroups が所持リストを同名で集約し、消費系（任意発動）を 先頭・続いてレアリティ降順に並べます。
  • 5件まで表示 ― HUDにはグループを5件まで。超過時は6行目に 「……」ボタンを出します。
  • 履歴ポップアップ ― 「……」クリックで、全所持アビリティを 横並びカードで一覧するポップアップを開きます。マウスホイールで左右スクロールでき、 消費系カードはクリックで発動できます。
ポップアップ表示中も時間は進む 履歴ポップアップは表示中もゲーム時間が進行します（盤面入力だけを止める）。 そのため、すぐ使いたい消費系アビリティはHUDの5枠に優先表示し、ポップアップを 開かなくても切れるようにしています。「UIを開いている間も時間が進む」という 緊張感のあるルールと、それに合わせたUI優先順位の設計です。
